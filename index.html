GitHub repókészítés

Bevezető

Amikor 2 vagy több ember egy projekten dolgozik a digitális térben, akkor valahova gyűjteni kell az 1-1 emberre leosztott részeit a munkának. Hogy ne kelljen feltöltögetni minden mentés után a felhőbe, vagy rosszabb esetben külső meghajtón elvinni személyesen a módosított információt, az informatikusok verziókezelőket alkalmaznak. A verziókezelés lényege, hogy otthon a kidolgozott feladatrészt fel lehet tölteni (push) egy távoli "mappába" (repository), amelyhez az összes tagnak hozzáférése van, és a módosításokat, javításokat le is lehet onnan "húzni" (pull), mindezt azért, hogy a legfrissebb fájlokkal tudjanak dolgozni. Így hatékonyan és viszonylag gyorsan be tudják fejezni a projektjüket.

A verziókezelés a Git nevű programmal zajlik, ugyanis ezzel lehet létrehozni a lokális, azaz helyi repókat. A helyi repók azért szükségesek, mert legelőször ebben kell lementeni a legelső fájlunkat, ezt tudjuk majd felküldeni a távoli repóba, továbbá ide kell menteni a módosított vagy újonnan hozzáadott fájlokat. A lokális repository készítésének és kezelésének a lépései a "Lokális repó kezelése" nevű gombra kattintva tekinthetőek meg.

Távoli repónak hívják azt a "mappát", amelyet a GitHub felületén is tudunk kezelni. Ez egy távoli hozzáférésű mappa, mindazonáltal saját parancssor vagy terminál használatával is konfigurálható. Ebbe kell "feltolnunk" a lokális repónkat, ha szeretnénk mások számára is láthatóvá, vagy akár módosíthatóvá tenni. A lehetőségek száma korlátlan, az opciók száma viszont nem. Tény, hogy rengeteg dolgot tudunk csinálni egy távoli repóval, viszont vannak olyan potenciálok, amelyhez a készítő engedélye is szükséges. Habár magától értetődő, hogy hatalmas mennyiségű munkától kíméli meg a felhasználóit a GitHub. A távoli repókészítés lépései a "Távoli repó kezelése" gomb megnyomásával tekinthetőek meg.

Lokális repó kezelése

Lokális repókészítés lépései

Első lépés: repository mappa meghatározása
Az első lépésben a lokális repónk mappáját kell meghatározni. Ez egy egyszerű folyamat, a mappában, amiben szeretnénk tárolni a fájljainkat, indítsunk el egy parancssort vagy jobb egér menüvel nyomjunk a "Git Bash Here" pontra (ehhez szükséges letölteni és feltelepíteni a Gitet). Továbbá megoldás lehet még a CLI kedvelőknek, hogy parancssorból vagy terminálból elnavigálnak a kívánt mappára. Ezek után git init paranccsal meghatározzuk, hogy melyik mappát szeretnénk repónak.
Második lépés: tegyünk a mappába néhány fájlt
A repónknak szüksége van fájlokra is, amiket el tud menteni. Húzzunk a mappába néhány fájlt, amiket később a távoli repónkba szeretnénk feltölteni vagy csak mentsünk a mappába párat. A fájlok bármilyen kiterjesztésűek és típusúak lehetnek.
Harmadik lépés: adjuk hozzá a fájlokat a színpadhoz
A színpad (stage) az a terület, ahonnan tudjuk commitelni a repó fájljait. A színpadhoz adás kihagyható rész, ugyanakkor nem ajánlott, mert nem kívánt változásokat eredményezhet.
Ha a színpadhoz szeretnénk adni fájlokat, azt 2 módon tehetjük meg. Az egyik módszer a git add fájlnév amellyel csak 1 db fájlt lehet egyszerre hozzáadni. A másik módszer a git add -A mely minden mappában lévő untracked (nyomon követhetetlen --> semmilyen művelet nem lett végrehajtva a fájlon) fájlt a színpadhoz ad, még akkor is, ha az már hozzá lett adva, de valamiért módosításra került. Egyszóval, minden fájlt a színpadhoz rendel.
Negyedik lépés: a commitelés
A commit egy checkpoint-ként szolgál. Ez azt jelenti, hogy mikor elérünk egy bizonyos szintig a projektünkben, érdemes commitelni minden fájlt, hogy egyrészt benne legyen a logban, másrészt vissza tudjunk térni ehhez a mentéshez, ha valami olyan módosítást hajtunk végre, ami a fájlon visszafordíthatatlan vagy nehezen visszafordítható. Commitelni csak azokat a fájlokat lehet, amelyek hozzá vannak adva a színpadhoz*. Automatikusan minden fájl "mentésre kerül", ha beírjuk a következő parancsot: git commit. Nem szükséges, ámbár erősen ajánlott üzenettel ellátni minden egyes commitelésünket, ugyanis ebből tudjuk meg, hogy melyik verziójuknál mentettük a fájlokat. Ezt a git commit -m "üzenet" paranccsal tudjuk megtenni.

*Nem ajánlott, bár a git commit -a -m "üzenet" parancs a színpadhoz adás folyamatát kihagyva commiteli az adatokat.
Ötödik lépés: kiegészítő parancsok
Mivel a lokális repó utolsó lehetséges lépése a commit, amivel végleges snapshotokat tudunk létrehozni a projekt különböző verzióiban, így most nézzük a kisegítő parancsokat, amik a munkához nem szügségesek, viszont nagy mértékben megtudják könnyíteni azt:
git status
- a fájlok és branchek állapotáról ad visszajelzést. A git status --short paranccsal egy kompaktabb nézet kérhető le, amelyben csak a fájlok és azok állapota látszik (?? - untracked, A - tracked, M - modified, D - deleted).

git help --all
- az összes gitben fellelhető parancsot és kapcsolóikat listázza ki, némi magyarázattal. Ellenben a git "parancs" -help csak a megadott parancshoz írja ki a tájékoztatót.

git --version
- az eszközre feltelepített Git verziószámát írja ki. Ha hibát jelez bármelyik git -tel kezdődő parancs, valószínűleg nincs feltelepítve a Git.
git log
- az előzetesen végrehajtott változ(tat)ásokat jeleníti meg.

Távoli repó kezelése

Távoli repó készítésének lépései

Első lépés: a lokális repó commitelése
Első lépésként fontos, hogy a helyi repónk összes fájlját a színpadra tegyük: git add -A majd commiteljük: git commit -m "üzenet".
Második lépés: a GitHub előkészítése
A következő lépésben a GitHub platformját kell előkészítenünk a lokális repó felküldéséhez. Ehhez be kell jelentkezni GitHubra, és mindenféle módosítás nélkül létre kell hoznunk egy repót GUI-ban. Miután megtörtént, ki kell másolnunk a repó URL-jét, majd a git remote add origin URL parancsot a helyi CLI-be beírva hozzá tudunk kapcsolódni a távoli repónkhoz abban az esetben*, ha a repó mappa kiválasztása után a CLI-ben is bejelentkezünk a Gitbe ugyanazzal az emaillel és felhasználónévvel, amellyel a GitHubba. A parancsa: git config --global user.name és git config --global user.email.

*A lépés kihagyható, viszont akkor a Git nem tudja kik vagyunk, így a commitelésekhez sem fog tudni információt csatolni, hogy ki hajtotta azokat végre.
Harmadik lépés: pusholás GitHubra
Ez egy egyszerű lépés. Mindösszesen annyi a teendőnk, hogy a lokális CLI-ben commiteljünk minden fájlt, ezután pedig a git push origin parancsot használva felküldjük az egész repónkat a távoli "mappába".
Negyedik lépés: pullozás GitHubról
Lehetőségünk van teljesen GUI-ban módosítani a repónkat a GitHub felületén pusholás után. Ha módosítottunk valamit az oldalon, és szeretnénk, hogy ezek a módosított fájlok a lokális repónkon is jelen legyenek vagy a csapattásunk által felküldött munkarészletet szeretnénk lementeni helyi meghajtóra a git pull origin parancsra lesz szükségünk. A pull a fetch és a merge parancs kombinációja, melyek lényegében beolvassák a távoli repón lévő összes adatot és szinkronizáják (vagy megalkotják) a helyi repóban.
Befejezés
Összefoglalva, a GitHub egy olyan platform, amely a Git alkalmazás segítségével gyárt "eszközöket", amelyek nagy mértékben segítik a digitális projekten dolgozó csapatok vagy csoportok munkáját. Használata egyáltalán nem bonyolult, mindösszesen elég pár parancsot ismernünk, hogy átlagos szinten tudjuk használni a verziókezelőt. A fentebb említett parancsokon és lépéseken kívül még lehet opcionális lépéseket is végrehajtani, de ezek egy átlagos projektmunkához elengedhetőek.
